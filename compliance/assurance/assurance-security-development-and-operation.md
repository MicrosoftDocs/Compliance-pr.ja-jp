<span data-ttu-id="c9f46-105">Microsoft のセキュリティ開発ライフサイクル (SDL) は、セキュリティで保護されたソフトウェアを開発および運用することに重点を置いたセキュリティ保証プロセスです。</span><span class="sxs-lookup"><span data-stu-id="c9f46-105">Microsoft's Security Development Lifecycle (SDL) is a security assurance process focused on developing and operating secure software.</span></span> <span data-ttu-id="c9f46-106">SDL には、Microsoft の開発者やエンジニアにとって、より多くの測定可能なセキュリティ要件が用意されています。 Microsoft は、お客様の製品とサービスの脆弱性の数と重大度を削減します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-106">The SDL provides detailed, measurable security requirements for developers and engineers at Microsoft to reduce the number and severity of vulnerabilities in our products and services.</span></span> <span data-ttu-id="c9f46-107">すべての Microsoft ソフトウェア開発チームは SDL の要件に従う必要があります。また、コンプライアンスのための脅威を変化させ、業界のベストプラクティスや規制基準を反映して SDL を継続的に更新しています。</span><span class="sxs-lookup"><span data-stu-id="c9f46-107">All Microsoft software development teams must follow SDL requirements, and we continuously update the SDL to reflect the changing threat landscape, industry best practices, and regulatory standards for compliance.</span></span>

<span data-ttu-id="c9f46-109">Microsoft の SDL プロセスは、要件、設計、実装、検証、リリースという5つの開発段階において考えることができます。</span><span class="sxs-lookup"><span data-stu-id="c9f46-109">The SDL process at Microsoft can be thought of in terms of five phases of development: requirements, design, implementation, verification, and release.</span></span> <span data-ttu-id="c9f46-110">まず、セキュリティを念頭に置いてソフトウェア要件を定義します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-110">It begins by defining software requirements with security in mind.</span></span> <span data-ttu-id="c9f46-111">これを行うには、アプリケーションが実行する必要があるセキュリティ関連の質問を確認します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-111">To do this, we ask security-relevant questions about what the application must accomplish.</span></span> <span data-ttu-id="c9f46-112">アプリケーションで機密データを収集する必要がありますか?</span><span class="sxs-lookup"><span data-stu-id="c9f46-112">Does the application need to collect sensitive data?</span></span> <span data-ttu-id="c9f46-113">アプリケーションは機密情報や重要なタスクを実行しますか?</span><span class="sxs-lookup"><span data-stu-id="c9f46-113">Will the application perform sensitive or important tasks?</span></span> <span data-ttu-id="c9f46-114">アプリケーションは、信頼できない発行元からの入力を受け入れる必要がありますか?</span><span class="sxs-lookup"><span data-stu-id="c9f46-114">Does the application need to accept input from untrusted sources?</span></span>

<span data-ttu-id="c9f46-115">関連するセキュリティ要件が特定されると、これらの要件を満たすセキュリティ機能が組み込まれるようにソフトウェアを設計します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-115">Once relevant security requirements have been identified, we design our software to incorporate security features that meet these requirements.</span></span> <span data-ttu-id="c9f46-116">開発者は SDL と設計要件をコードに実装し、手動のコード レビュー、自動化されたセキュリティ ツール、および侵入テストで検証します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-116">Our developers implement SDL and design requirements in the code, which we verify through manual code review, automated security tooling, and penetration testing.</span></span> <span data-ttu-id="c9f46-117">最後に、コードをリリースする前に、最終的なセキュリティとプライバシーに関するレビューを行って、すべての要件を満たしていることを確認してください。</span><span class="sxs-lookup"><span data-stu-id="c9f46-117">Finally, before code can be released, new features and material changes undergo final security and privacy review to ensure all requirements are met.</span></span>

<span data-ttu-id="c9f46-119">コード開発やリリース後のセキュリティ要件を実装する開発者をサポートするために、Microsoft では、セキュリティの欠陥や脆弱性についてソース コードを自動的にチェックする、セキュリティで保護された一連の開発ツールを提供しています。</span><span class="sxs-lookup"><span data-stu-id="c9f46-119">To support our developers in implementing security requirements during code development and after release, Microsoft provides a suite of secure development tools to automatically check source code for security flaws and vulnerabilities.</span></span> <span data-ttu-id="c9f46-120">Microsoft は、Microsoft build パイプライン内で自動的に実行される組み込みのセキュリティチェックとともに、使用する開発者向けの承認済みツール (コンパイラ、開発環境など) のリストを定義して公開します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-120">Microsoft defines and publishes a list of approved tools for our developers to use, such as compilers and development environments, along with the built-in security checks executed automatically within Microsoft build pipelines.</span></span> <span data-ttu-id="c9f46-121">弊社の開発者は、新しいセキュリティ機能を活用するために、承認されたツールの最新バージョンを使用しています。</span><span class="sxs-lookup"><span data-stu-id="c9f46-121">Our developers use the latest versions of approved tools to take advantage of new security features.</span></span>

<span data-ttu-id="c9f46-122">コードをリリースブランチにチェックインする前に、SDL では別のレビュー担当者が手動でコードレビューを行う必要があります。</span><span class="sxs-lookup"><span data-stu-id="c9f46-122">Before code can be checked into a release branch, the SDL requires manual code review by a separate reviewer.</span></span> <span data-ttu-id="c9f46-123">コード レビュー担当者は、コーディング エラーを確認し、コード変更が SDL および設計要件を満たしているか、機能およびセキュリティのテストに合格したか、確実に動作するかを確認します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-123">Code reviewers check for coding errors and verify that code changes meet SDL and design requirements, pass functional and security tests, and perform reliably.</span></span> <span data-ttu-id="c9f46-124">また、コードの変更が適切に文書化され、意図しない副作用が発生しないことを確認するために、関連するドキュメント、構成、および依存関係を確認します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-124">They also review associated documentation, configs, and dependencies to ensure code changes are documented appropriately and will not cause unintended side-effects.</span></span> <span data-ttu-id="c9f46-125">レビュー担当者は、コード レビュー中に問題を発見した場合、提出者に、提案された変更と追加のテストを含めたコードを再提出するよう依頼できます。</span><span class="sxs-lookup"><span data-stu-id="c9f46-125">If a reviewer finds problems during code review, they can ask the submitter to resubmit the code with suggested changes and additional testing.</span></span> <span data-ttu-id="c9f46-126">また、コード担当者は、要件を満たしていないコードのチェックインを全面的にブロックすることもできます。</span><span class="sxs-lookup"><span data-stu-id="c9f46-126">Code reviewers may also decide to block check-in entirely for code that does not meet requirements.</span></span> <span data-ttu-id="c9f46-127">レビュー担当者がコードを満足していると判断されると、レビュー担当者は承認を提供します。これは、コードが次の展開フェーズに進む前に必要になります。</span><span class="sxs-lookup"><span data-stu-id="c9f46-127">Once the code has been deemed satisfactory by the reviewer, the reviewer provides approval, which is required before code can proceed to the next deployment phase.</span></span>

<span data-ttu-id="c9f46-128">セキュリティで保護された開発ツールと手動のコードレビューに加えて、Microsoft は自動セキュリティツールを使用して SDL の要件を適用します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-128">In addition to secure development tools and manual code review, Microsoft enforces SDL requirements using automated security tooling.</span></span> <span data-ttu-id="c9f46-129">これらのツールの多くは、コミットパイプラインに組み込まれており、セキュリティ上の問題がチェックインされ、新しいビルドがコンパイルされたときに、そのコードを自動的に分析します。</span><span class="sxs-lookup"><span data-stu-id="c9f46-129">Many of these tools are built into the commit pipeline and automatically analyze code for security flaws as it is checked in and as new builds are compiled.</span></span> <span data-ttu-id="c9f46-130">例としては、一般的なセキュリティの欠陥と、埋め込まれた機密情報のコードを分析する資格情報スキャナーの静的コード分析があります。</span><span class="sxs-lookup"><span data-stu-id="c9f46-130">Examples include static code analysis for common security flaws and credential scanners that analyze code for embedded secrets.</span></span> <span data-ttu-id="c9f46-131">新しいビルドがセキュリティレビューを通過してリリースを承認されるようにするには、自動セキュリティツールによって検出された問題を修正する必要があります。</span><span class="sxs-lookup"><span data-stu-id="c9f46-131">Issues uncovered by automated security tools must be fixed before new builds can pass security review and be approved for release.</span></span>
