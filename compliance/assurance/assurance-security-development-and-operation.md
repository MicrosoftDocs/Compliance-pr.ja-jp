<span data-ttu-id="cee00-105">Microsoft のセキュリティ開発ライフサイクル (SDL) は、セキュリティ保護されたソフトウェアの開発と運用に重点を置くセキュリティ 保証プロセスです。</span><span class="sxs-lookup"><span data-stu-id="cee00-105">Microsoft's Security Development Lifecycle (SDL) is a security assurance process focused on developing and operating secure software.</span></span> <span data-ttu-id="cee00-106">SDL には、Microsoft の開発者やエンジニアにとって、より多くの測定可能なセキュリティ要件が用意されています。 Microsoft は、お客様の製品とサービスの脆弱性の数と重大度を削減します。</span><span class="sxs-lookup"><span data-stu-id="cee00-106">The SDL provides detailed, measurable security requirements for developers and engineers at Microsoft to reduce the number and severity of vulnerabilities in our products and services.</span></span> <span data-ttu-id="cee00-107">すべての Microsoft ソフトウェア開発チームは、SDL の要件に従う必要があります。また、変化する脅威の状況、業界のベスト プラクティス、コンプライアンスに関する規制基準を反映するように SDL を継続的に更新します。</span><span class="sxs-lookup"><span data-stu-id="cee00-107">All Microsoft software development teams must follow SDL requirements, and we continuously update the SDL to reflect the changing threat landscape, industry best practices, and regulatory standards for compliance.</span></span>

## <a name="how-does-microsofts-sdl-improve-application-security"></a><span data-ttu-id="cee00-108">Microsoft の SDL は、アプリケーションのセキュリティを向上させる方法を説明します。</span><span class="sxs-lookup"><span data-stu-id="cee00-108">How does Microsoft's SDL improve application security?</span></span>

<span data-ttu-id="cee00-109">Microsoft の SDL プロセスは、要件、設計、実装、検証、およびリリースの 5 つのフェーズで考え得る。</span><span class="sxs-lookup"><span data-stu-id="cee00-109">The SDL process at Microsoft can be thought of in terms of five phases of development: requirements, design, implementation, verification, and release.</span></span> <span data-ttu-id="cee00-110">まず、セキュリティを念頭に置いてソフトウェア要件を定義します。</span><span class="sxs-lookup"><span data-stu-id="cee00-110">It begins by defining software requirements with security in mind.</span></span> <span data-ttu-id="cee00-111">これを行うには、アプリケーションが実行する必要があるセキュリティ関連の質問を確認します。</span><span class="sxs-lookup"><span data-stu-id="cee00-111">To do this, we ask security-relevant questions about what the application must accomplish.</span></span> <span data-ttu-id="cee00-112">アプリケーションで機密データを収集する必要がありますか?</span><span class="sxs-lookup"><span data-stu-id="cee00-112">Does the application need to collect sensitive data?</span></span> <span data-ttu-id="cee00-113">アプリケーションは機密情報や重要なタスクを実行しますか?</span><span class="sxs-lookup"><span data-stu-id="cee00-113">Will the application perform sensitive or important tasks?</span></span> <span data-ttu-id="cee00-114">アプリケーションは、信頼できない発行元からの入力を受け入れる必要がありますか?</span><span class="sxs-lookup"><span data-stu-id="cee00-114">Does the application need to accept input from untrusted sources?</span></span>

<span data-ttu-id="cee00-115">関連するセキュリティ要件が特定されると、これらの要件を満たすセキュリティ機能が組み込まれるようにソフトウェアを設計します。</span><span class="sxs-lookup"><span data-stu-id="cee00-115">Once relevant security requirements have been identified, we design our software to incorporate security features that meet these requirements.</span></span> <span data-ttu-id="cee00-116">開発者は SDL と設計要件をコードに実装し、手動のコード レビュー、自動化されたセキュリティ ツール、および侵入テストで検証します。</span><span class="sxs-lookup"><span data-stu-id="cee00-116">Our developers implement SDL and design requirements in the code, which we verify through manual code review, automated security tooling, and penetration testing.</span></span> <span data-ttu-id="cee00-117">最後に、コードをリリースする前に、新しい機能と重要な変更は最終的なセキュリティとプライバシーのレビューを受け、すべての要件が満たされます。</span><span class="sxs-lookup"><span data-stu-id="cee00-117">Finally, before code can be released, new features and material changes undergo final security and privacy review to ensure all requirements are met.</span></span>

## <a name="how-does-microsoft-test-source-code-for-common-vulnerabilities"></a><span data-ttu-id="cee00-118">Microsoft は、一般的な脆弱性のソース コードをテストする方法を説明します。</span><span class="sxs-lookup"><span data-stu-id="cee00-118">How does Microsoft test source code for common vulnerabilities?</span></span>

<span data-ttu-id="cee00-119">コード開発やリリース後のセキュリティ要件を実装する開発者をサポートするために、Microsoft では、セキュリティの欠陥や脆弱性についてソース コードを自動的にチェックする、セキュリティで保護された一連の開発ツールを提供しています。</span><span class="sxs-lookup"><span data-stu-id="cee00-119">To support our developers in implementing security requirements during code development and after release, Microsoft provides a suite of secure development tools to automatically check source code for security flaws and vulnerabilities.</span></span> <span data-ttu-id="cee00-120">Microsoft は、コンパイラや開発環境など、開発者が使用する承認済みツールの一覧と、Microsoft ビルド パイプライン内で自動的に実行される組み込みのセキュリティ チェックを定義して発行します。</span><span class="sxs-lookup"><span data-stu-id="cee00-120">Microsoft defines and publishes a list of approved tools for our developers to use, such as compilers and development environments, along with the built-in security checks executed automatically within Microsoft build pipelines.</span></span> <span data-ttu-id="cee00-121">弊社の開発者は、新しいセキュリティ機能を活用するために、承認されたツールの最新バージョンを使用しています。</span><span class="sxs-lookup"><span data-stu-id="cee00-121">Our developers use the latest versions of approved tools to take advantage of new security features.</span></span>

<span data-ttu-id="cee00-122">コードをリリース ブランチにチェックインする前に、SDL には別のレビューアーによる手動のコード レビューが必要です。</span><span class="sxs-lookup"><span data-stu-id="cee00-122">Before code can be checked into a release branch, the SDL requires manual code review by a separate reviewer.</span></span> <span data-ttu-id="cee00-123">コード レビュー担当者は、コーディング エラーを確認し、コード変更が SDL および設計要件を満たしているか、機能およびセキュリティのテストに合格したか、確実に動作するかを確認します。</span><span class="sxs-lookup"><span data-stu-id="cee00-123">Code reviewers check for coding errors and verify that code changes meet SDL and design requirements, pass functional and security tests, and perform reliably.</span></span> <span data-ttu-id="cee00-124">また、コードの変更が適切に文書化され、意図しない副作用が発生しないことを確認するために、関連するドキュメント、構成、および依存関係を確認します。</span><span class="sxs-lookup"><span data-stu-id="cee00-124">They also review associated documentation, configs, and dependencies to ensure code changes are documented appropriately and will not cause unintended side-effects.</span></span> <span data-ttu-id="cee00-125">レビュー担当者は、コード レビュー中に問題を発見した場合、提出者に、提案された変更と追加のテストを含めたコードを再提出するよう依頼できます。</span><span class="sxs-lookup"><span data-stu-id="cee00-125">If a reviewer finds problems during code review, they can ask the submitter to resubmit the code with suggested changes and additional testing.</span></span> <span data-ttu-id="cee00-126">また、コード担当者は、要件を満たしていないコードのチェックインを全面的にブロックすることもできます。</span><span class="sxs-lookup"><span data-stu-id="cee00-126">Code reviewers may also decide to block check-in entirely for code that does not meet requirements.</span></span> <span data-ttu-id="cee00-127">レビュー者によってコードが満足できると判断された後、レビューアーは承認を提供します。これは、コードが次の展開フェーズに進む前に必要です。</span><span class="sxs-lookup"><span data-stu-id="cee00-127">Once the code has been deemed satisfactory by the reviewer, the reviewer provides approval, which is required before code can proceed to the next deployment phase.</span></span>

<span data-ttu-id="cee00-128">Microsoft では、セキュリティ保護された開発ツールと手動コード レビューに加えて、自動セキュリティ ツールを使用して SDL 要件を適用します。</span><span class="sxs-lookup"><span data-stu-id="cee00-128">In addition to secure development tools and manual code review, Microsoft enforces SDL requirements using automated security tooling.</span></span> <span data-ttu-id="cee00-129">これらのツールの多くは、コミット パイプラインに組み込まれています。</span><span class="sxs-lookup"><span data-stu-id="cee00-129">Many of these tools are built into the commit pipeline and automatically analyze code for security flaws as it is checked in and as new builds are compiled.</span></span> <span data-ttu-id="cee00-130">たとえば、一般的なセキュリティ上の欠陥に対する静的なコード分析や、埋め込みシークレットのコードを分析する資格情報スキャナーがあります。</span><span class="sxs-lookup"><span data-stu-id="cee00-130">Examples include static code analysis for common security flaws and credential scanners that analyze code for embedded secrets.</span></span> <span data-ttu-id="cee00-131">自動セキュリティ ツールによって発見された問題は、新しいビルドがセキュリティ レビューに合格し、リリースが承認される前に修正する必要があります。</span><span class="sxs-lookup"><span data-stu-id="cee00-131">Issues uncovered by automated security tools must be fixed before new builds can pass security review and be approved for release.</span></span>

<span data-ttu-id="cee00-133">Microsoft では、次の目的で設計されたツールとワークフローを活用する、オープンソース のセキュリティを管理するための高度な戦略を採用しています。</span><span class="sxs-lookup"><span data-stu-id="cee00-133">Microsoft has adopted a high-level strategy for managing open-source security, which leverages tools and workflows designed to:</span></span>
