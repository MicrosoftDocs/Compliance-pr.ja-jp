# <a name="microsoft-365-security-incident-management-preparation"></a>Microsoft 365 セキュリティ インシデント管理: 準備

## <a name="training-and-background-checks"></a>トレーニングとバックグラウンド チェック

Microsoft 365 に取り組む各従業員には、その役割に適したセキュリティ インシデントと対応手順に関するトレーニングが提供されます。 すべての Microsoft 365 従業員は、参加時にトレーニングを受け、その後は毎年年次リフレッシュ トレーニングを受ける。 このトレーニングは、トレーニングが完了すると、すべての従業員が以下を理解するように、セキュリティに対する Microsoft のアプローチの基本的な理解を従業員に提供するように設計されています。

- セキュリティ インシデントの定義
- すべての従業員がセキュリティ インシデントを報告する責任
- 潜在的なセキュリティ インシデントを Microsoft 365 セキュリティ対応チームにエスカレートする方法
- Microsoft 365 セキュリティ インシデント対応チームがセキュリティ インシデントに対応する方法
- プライバシー、特に顧客のプライバシーに関する特別な懸念事項
- セキュリティとプライバシー、エスカレーション連絡先に関する追加情報の検索先
- その他の関連するセキュリティ領域 (必要に応じて)

適切な従業員は、毎年セキュリティに関する更新トレーニングを受ける。 年次更新プログラムのトレーニングでは、次の分野に重点を当ててはっています。

- 前の年に標準の運用手順に加えた変更
- すべてのユーザーがセキュリティ インシデントを報告する責任と、その方法
- セキュリティとプライバシー、エスカレーション連絡先に関する追加情報の検索先
- 毎年関連する可能性があるその他のセキュリティフォーカス分野

また、Microsoft 365 に取り組む各従業員は、医療保険の移植性と責任に関する法律 (HIPAA)、武器国際取引規則 (ITAR)、連邦リスク/承認管理プログラム (FedRAMP) などの米国の規制に従って、求職者の教育、雇用、犯罪履歴、その他の特定の情報を含む適切で徹底的な身元調査を行います。

バックグラウンド チェックは、Microsoft 365 エンジニアリングで働くすべての従業員に必須です。 一部の Microsoft 365 環境とオペレーターの役割では、完全な指紋認証、市民権の要件、政府による権限要件、その他のより厳しい管理が必要になる場合があります。 また、一部のサービス チームと役割は、必要に応じて特殊なセキュリティ トレーニングを受け取る場合があります。 最後に、セキュリティ チームのメンバー自身が、セキュリティに直接関連する特別なトレーニングと会議への参加を取得します。 このトレーニングは、チームと従業員の必要によって異なりますが、業界の既知のセキュリティ トレーニング ベンダーによる業界の会議、内部の Microsoft セキュリティ会議、外部トレーニング コースが含まれます。 また、Microsoft 全体のセキュリティ コミュニティ向けとして 1 年を通して公開され、Microsoft 365 に特化した専用のセキュリティ トレーニング記事も用意されています。

## <a name="penetration-testing--assessment"></a>侵入テストと&評価

Microsoft は、さまざまな業界団体やセキュリティの専門家と一緒に、新しい脅威と進化する傾向を理解しています。 Microsoft は、脆弱性について独自のシステムを継続的に評価し、同じことを行うさまざまな独立した外部の専門家と契約します。

Microsoft 365 内でサービスの強化を行うテストは、次の 4 つの一般的なカテゴリにグループ化できます。

1. **自動セキュリティ テスト:** 内部および外部の担当者は、Microsoft SDL プラクティス、Open Web Application Security Project (OWASP) の上位 10 件のリスク、およびさまざまな業界から報告された新たな脅威に基づいて、Microsoft 365 環境を定期的にスキャンします。
2. **脆弱性評価:** 独立したサードパーティのテスターとの正式な契約は、さまざまな規制機関のサービス義務を果たすために主要な論理コントロールが効果的に動作するかどうかを定期的に検証します。 評価は、OWASP Top 10 リスクおよび他のサービスに適用される脅威に基づいて、CREST (Registered Ethical Security Testers) 認定の担当者によって実施されます。 検出された脅威はすべて、クローズに追跡されます。
3. **継続的なシステム脆弱性テスト:** Microsoft は定期的なテストを実行します。このテストでは、新たな脅威、ブレンドされた脅威、高度な持続的な脅威を使用してチームがシステムを侵害しようとする一方で、他のチームはそのような侵害の試みをブロックします。
4. **Microsoft Online Services プログラム**: このプログラムは、Microsoft 365 で限定的な、お客様が作成した脆弱性評価を許可するポリシーを運用します。 詳細については、「Bug [B Microsoft Online Services Terms 」を参照してください](https://www.microsoft.com/msrc/bounty-terms)。

Microsoft 365 エンジニアリング チームは、さまざまなコンプライアンス ドキュメントを定期的に公開します。 これらのドキュメントのいくつかは[、Microsoft Cloud Service Trust Portal](https://aka.ms/STP)または Microsoft [365](https://compliance.office.com)コンプライアンス センターのサービス アシュアランス領域から、情報開示契約の下で利用できます。

>[!NOTE]
>Service Trust Portal にアクセスする方法の詳細については、「Office 365 for business、Azure、Dynamics CRM Online サブスクリプションの Service Trust Portal の使用を開始する」を参照してください。 Microsoft 365 コンプライアンス センターにアクセスするには、Microsoft 365 サブスクリプションが必要です。

## <a name="attack-simulation"></a>攻撃シミュレーション

Microsoft では、継続的な攻撃シミュレーション演習と、セキュリティおよび対応計画の実地侵入テストに取り組み、検出および対応能力の向上を目的とします。 Microsoft は、現実世界の侵害を定期的にシミュレートし、継続的なセキュリティ監視を実施し、セキュリティ インシデント対応を実施して、Microsoft 365 と Azure の両方のセキュリティを検証および改善します。

Microsoft は、次の 2 つの主要なチームを使用して、侵害のセキュリティ戦略を実行します。

### <a name="red-teams"></a>レッド チーム

Microsoft 365 セキュリティ レッド チームは、Microsoft のインフラストラクチャ、プラットフォーム、および Microsoft 独自のテナントとアプリケーションの侵害に重点を当てた、Microsoft 内のフルタイム スタッフのグループです。 これらは、オンライン サービスに対して標的型かつ持続的な攻撃を実行する専用の敵対者 (倫理的ハッカーのグループ) です (ただし、顧客のアプリケーションやデータは実行しない)。 サービス インシデント対応機能の継続的な "フル スペクトル" 検証 (技術コントロール、紙のポリシー、人間の対応など) を提供します。

### <a name="blue-teams"></a>ブルー チーム

Microsoft 365 セキュリティ ブルー チームは、セキュリティ インシデント対応、エンジニアリング、運用チーム全体のセキュリティレスポンダーとメンバーの専用セットで構成されています。 これらは独立したチームであり、レッド チームとは別に運用しています。 Blue Team は確立されたセキュリティ プロセスに従い、最新のツールとテクノロジを使用して攻撃や侵入の試みを検出し、対応します。 実際の攻撃と同様に、ブルー チームはレッド チームの攻撃がいつ、どのように発生するか、どのような方法が使用されるのかを知りません。 レッド チーム攻撃でも実際の攻撃でも、彼らの仕事は、すべてのセキュリティ インシデントを検出して対応します。 このため、ブルー チームは継続的にオンコールを行い、レッド チームが他の敵対者と同じ方法で違反した場合に対応する必要があります。

Microsoft は、サービス全体とサービス内の両方で運用を行うさまざまな部門で、赤のフルタイム チームと青色のチームを Microsoft 全体で分離しています。 レッド チームと呼ばれるこのアプローチは、実際の運用インフラストラクチャに対して、実際の運用インフラストラクチャに対して、実際の敵対者と同じ手法、手法、および手順を使用して、Microsoft サービスのシステムと運用全体をテストし、インフラストラクチャおよびプラットフォームエンジニアリングチームまたは運用チームの詳細を確認しない方法です。 これにより、セキュリティの検出と対応の機能をテストし、制御された方法で、実稼働の脆弱性、構成エラー、無効な想定、その他のセキュリティの問題を特定するのに役立ちます。 すべてのレッド チーム違反の後に、サービス チームを含むレッド チームとブルー チームの間で完全に開示され、ギャップを特定し、発見に対処し、違反への対応を大幅に改善します。

>[!NOTE]
>Red Teaming またはライブ サイト侵入の演習中は、顧客データを対象となさいません。 テストは、Microsoft 365 と Azure のインフラストラクチャとプラットフォーム、および Microsoft 独自のテナント、アプリケーション、およびデータに対して行います。 Microsoft 365 または Azure でホストされている顧客テナント、アプリケーション、およびデータは、契約のルールに同意した場合に対象とされません。

### <a name="joint-exercises"></a>共同演習

Microsoft 365 Security Blue チームと Red チームは、運用中の関係が各チームの選択した従業員との敵対よりもパートナーである共同運用を行う場合があります。 これらの演習は、倫理的ハッカーとレスポンダーの間のリアルタイムコラボレーションを通じて、よりターゲットを絞った結果のセットを推進するために、チーム間で十分に調整されています。 これらの 「紫色のチーム」演習は、機会を最大限に活用するために各操作に合わせて高度に調整されています。ただし、各操作の基本は、目的を達成するための高帯域幅の情報共有とパートナーシップです。
