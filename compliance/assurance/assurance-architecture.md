## <a name="how-does-microsoft-365-ensure-isolation-between-customer-tenants"></a><span data-ttu-id="d9941-114">Microsoft 365 では、顧客テナント間の分離をどのようにしていますか?</span><span class="sxs-lookup"><span data-stu-id="d9941-114">How does Microsoft 365 ensure isolation between customer tenants?</span></span>

<span data-ttu-id="d9941-115">Microsoft のクラウドサービスは、すべてのテナントが他のすべてのテナントに敵意を与える可能性があることを前提として構築されています。</span><span class="sxs-lookup"><span data-stu-id="d9941-115">Microsoft's cloud services are built on the assumption that all tenants are potentially hostile to all other tenants.</span></span> <span data-ttu-id="d9941-116">テナントを相互に適切に分離するために、Microsoft はさまざまな分離テクノロジと制御を実装しています。</span><span class="sxs-lookup"><span data-stu-id="d9941-116">To properly isolate tenants from one another, Microsoft implements a variety of isolation technologies and controls.</span></span> <span data-ttu-id="d9941-117">これらのコントロールは、テナント間での情報漏洩または承認されていない顧客データへのアクセスを防止するために設計されており、1つのテナントのアクションが別のテナントのサービスに悪影響を及ぼすことを防ぐためのものです。</span><span class="sxs-lookup"><span data-stu-id="d9941-117">These controls are designed to safeguard against information leakage or unauthorized access to customer data across tenants and to prevent the actions of one tenant from adversely affecting the service for another tenant.</span></span>

<span data-ttu-id="d9941-118">お客様のコンテンツは、Microsoft 365 テナント内の Azure Active Directory (Azure AD) を使用して論理的に分離されています。</span><span class="sxs-lookup"><span data-stu-id="d9941-118">Customer content is logically isolated within Microsoft 365 tenants using Azure Active Directory (Azure AD).</span></span> <span data-ttu-id="d9941-119">Microsoft 365 のユーザー認証は、ユーザー ID を確認するだけでなく、ユーザー アカウントが含まれるテナントの ID も確認します。これにより、ユーザーはテナント環境外のデータにアクセスできなくなります。</span><span class="sxs-lookup"><span data-stu-id="d9941-119">User authentication in Microsoft 365 verifies not only the user identity, but also the tenant identity the user account is part of, preventing users from accessing data outside their tenant environment.</span></span> <span data-ttu-id="d9941-120">Azure AD の論理的分離を補足するために、お客様のコンテンツは常に保存され、転送中で暗号化されます。</span><span class="sxs-lookup"><span data-stu-id="d9941-120">To supplement the logical isolation of Azure AD, customer content is always encrypted at rest and in transit.</span></span> <span data-ttu-id="d9941-121">個別のサービスによって、テナント分離の追加層が提供される場合もあります。たとえば、テナントデータの SharePoint Online 分離が、暗号化された個別のデータベースに分かれています。</span><span class="sxs-lookup"><span data-stu-id="d9941-121">Individual services may also provide additional layers of tenant isolation, such as SharePoint Online isolation of tenant data in separate, encrypted databases.</span></span>

## <a name="how-does-microsoft-365-engineer-resilient-services-that-avoid-single-points-of-failure"></a><span data-ttu-id="d9941-122">Microsoft 365 は、単一点障害を回避するための復元性の高いサービスを示していますか?</span><span class="sxs-lookup"><span data-stu-id="d9941-122">How does Microsoft 365 engineer resilient services that avoid single points of failure?</span></span>

<span data-ttu-id="d9941-123">Microsoft は、クラウドサービスを設計および構築して信頼性を最大化し、通常の運用における障害や課題の影響をお客様にもたらす悪影響を最小限に抑えます。</span><span class="sxs-lookup"><span data-stu-id="d9941-123">Microsoft designs and builds cloud services to maximize reliability and minimize the negative effects on customers in the face of faults and challenges to normal operations.</span></span> <span data-ttu-id="d9941-124">この戦略は、地理的に分散したデータセンターを接続するネットワークの設計から始まります。</span><span class="sxs-lookup"><span data-stu-id="d9941-124">This strategy begins with the design of the network connecting our geographically distributed datacenters.</span></span> <span data-ttu-id="d9941-125">Microsoft のネットワークアーキテクチャには、直接 interconnections と複数のネットワークパスが含まれています。</span><span class="sxs-lookup"><span data-stu-id="d9941-125">Microsoft's network architecture includes direct interconnections and multiple network paths.</span></span> <span data-ttu-id="d9941-126">Microsoft 365 サービスでは、この冗長性を活用して、サービス品質を向上するために障害を回避するトラフィックを自動的にルーティングします。</span><span class="sxs-lookup"><span data-stu-id="d9941-126">Microsoft 365 services leverage this redundancy to automatically route traffic around failures to improve service quality.</span></span>

<span data-ttu-id="d9941-127">サービスレベルでは、Microsoft 365 の復元戦略によってソフトウェアの復元性が優先されます。</span><span class="sxs-lookup"><span data-stu-id="d9941-127">At the service level, Microsoft 365's resilience strategy prioritizes software resiliency.</span></span> <span data-ttu-id="d9941-128">可能な限り、サービスの正常性監視を使用してサービスをアクティブ/アクティブ構成に展開することで、ユーザーの介入なしに多くの一般的な障害やエラーを検出し、回復することができます。</span><span class="sxs-lookup"><span data-stu-id="d9941-128">Wherever possible, our services are deployed in active/active configurations with automated service health monitoring, allowing the service to detect and recover from many common faults and failures without human intervention.</span></span> <span data-ttu-id="d9941-129">アクティブ/アクティブ構成に加えて、Microsoft 365 サービスは、サービスが別のフォールトゾーンに展開されるようにすることにより、フォールトトレランスが向上し、1つの領域内の障害が他の領域の可用性に影響を与えることを防ぐことができます。</span><span class="sxs-lookup"><span data-stu-id="d9941-129">In addition to active/active configurations, Microsoft 365 services increase fault-tolerance by ensuring the service is deployed in separate fault zones, preventing a fault in one zone from affecting the availability of other zones.</span></span>

<span data-ttu-id="d9941-130">データの回復性は、Microsoft 365 サービスのデータの整合性と可用性を保護することにより、サービスの回復性を補完します。</span><span class="sxs-lookup"><span data-stu-id="d9941-130">Data resiliency complements service resiliency by protecting the integrity and availability of data in Microsoft 365 services.</span></span> <span data-ttu-id="d9941-131">サービスでは、ローカル記憶域の冗長性と geo 冗長を使用して、顧客データのコピーを別の障害ゾーンにレプリケートします。</span><span class="sxs-lookup"><span data-stu-id="d9941-131">Our services use local storage redundancy and geo-redundancy to replicate copies of customer data into different fault zones.</span></span> <span data-ttu-id="d9941-132">ある障害ゾーンでデータが破損または失われた場合、可用性を失うことなく別の障害ゾーンでデータにアクセスできます。</span><span class="sxs-lookup"><span data-stu-id="d9941-132">If data is corrupted or lost in one fault zone, it can be accessed in another fault zone without loss of availability.</span></span> <span data-ttu-id="d9941-133">自動整合性チェックは、多くの種類の物理的または論理的な破損によって影響を受けるデータを自動的に復元します。</span><span class="sxs-lookup"><span data-stu-id="d9941-133">Automated integrity checking automatically restores data impacted by many kinds of physical or logical corruption.</span></span> <span data-ttu-id="d9941-134">また、Microsoft 365 は、Exchange Online および SharePoint Online でユーザーが誤って削除または変更したデータを復元するためのツールをお客様に提供します。</span><span class="sxs-lookup"><span data-stu-id="d9941-134">Microsoft 365 also provides customers with tools to restore data accidentally deleted or modified by the customer in Exchange Online and SharePoint Online.</span></span>

<span data-ttu-id="d9941-140">Microsoft 365 アーキテクチャは定期的に検証されており、クラウドベースのツールを使用して自動的に自動的に使用し、セキュリティ原則に沿って配置し、分離と復元の機能を継続的にテストしています。</span><span class="sxs-lookup"><span data-stu-id="d9941-140">Microsoft 365 architecture is validated regularly and automatically using cloud-based tools to verify alignment with our security principles and to continuously test isolation and resiliency features.</span></span> <span data-ttu-id="d9941-141">アーキテクチャ検証は、目的の状態からサービスの現在の状態が drifted されているインスタンスを自動的に識別し、レビューと軽減のための逸脱を指摘します。</span><span class="sxs-lookup"><span data-stu-id="d9941-141">Architectural validation works to automatically identify instances where the current state of the service has drifted from the desired state, flagging any deviations for review and mitigation.</span></span> <span data-ttu-id="d9941-142">アーキテクチャ検証の目的は、サービスインフラストラクチャのセキュリティ機能が期待どおりに機能し続けるようにすることです。</span><span class="sxs-lookup"><span data-stu-id="d9941-142">The goal of architecture validation is to ensure the security capabilities of our service infrastructure continue to function as expected.</span></span>
